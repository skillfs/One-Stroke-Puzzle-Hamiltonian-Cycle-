## 1. บทนำและขอบเขต

เกม One-Stroke Puzzle ที่ศึกษาในวิทยานิพนธ์นี้ มีกติกาที่นิยามได้ชัดเจนว่า
**"ผู้เล่นลากเส้นเดียวผ่านทุกช่องที่เล่นได้จนครบ โดยห้ามผ่านช่อง (จุด) เดิมซ้ำ"**
ซึ่งในเชิงทฤษฎีกราฟ (Graph Theory) ตรงกับการหา **Hamiltonian Path** บนกราฟที่เกิดจากช่องของด่าน

วัตถุประสงค์ของช่วงงานนี้คือ
1. ทดสอบว่า Backbite Algorithm (ที่ออกแบบ pseudocode ไว้แล้ว) ใช้งานได้จริง
2. สำรวจแนวทาง algorithm สำหรับ "สร้างด่านอัตโนมัติ" ที่ให้ด่านมีคุณสมบัติตามต้องการ 3 ประการ:
   - **(ก)** รูปทรงด่านไม่เป็นก้อนทึบ (ไม่เป็น blob)
   - **(ข)** มีช่องว่าง (space) แทรกอยู่ภายในด่าน
   - **(ค)** มี "ทางตันลวง" (trap / dead end) ให้ผู้เล่นเดินผิดได้จำนวนมาก

## 2. สิ่งที่ลงมือพัฒนาจริง (Implementation)

### 2.1 เครื่องมือทดสอบเชิงโต้ตอบ (Interactive Visualizer)

พัฒนาเครื่องมือทดสอบเป็นไฟล์ HTML/JavaScript แบบ self-contained
(`backbite_visualizer.html`) เพื่อใช้ยืนยันว่า algorithm ทำงานถูกต้องจริง
ไม่ใช่แค่บน pseudocode โดยมีความสามารถ:

- เลือกขนาด Grid (3×3 ถึง 8×8)
- ปรับ Quality Factor (q) ของ Backbite
- กำหนด Seed เพื่อ reproduce ผลลัพธ์เดิมได้ (ใช้ Mulberry32 PRNG)
- แสดงผล path บน Canvas พร้อม animation
- แสดงค่าสถิติ (Metrics) ของด่านที่สร้าง

### 2.2 Backbite Algorithm + Tail-Cutting (โหมด Single Grid)

- ยืนยันแล้วว่า Backbite สร้าง Hamiltonian Path ได้ครบทุกช่องโดยไม่ซ้ำ
  (ทดสอบ 5×5 = 25 ช่อง, 6×6 = 36 ช่อง ผ่านทั้งหมด)
- เพิ่มกลไก **Tail-Cutting**: สร้าง path เต็มก่อน แล้วตัดส่วนปลาย (tail) ออก
  ตามอัตราส่วน Fill% ส่วนที่ตัดออกกลายเป็นช่องว่าง (holes)
- ผลทดสอบ (verified): 6×6 ที่ Fill 70% ได้ 25 ช่องเล่นได้ + 11 ช่องว่าง = 36 สมดุล

### 2.3 Chain Mode (รวมหลาย Sub-Path เป็นเส้นเดียว)

- สร้าง Hamiltonian Path ย่อยหลายก้อน (รองรับ block สี่เหลี่ยมผืนผ้า)
  แล้วเชื่อมต่อปลายของก้อนหนึ่ง (tail) เข้ากับต้นของก้อนถัดไป (head)
  ผ่าน "bridge cell" ให้กลายเป็นเส้นเดียวต่อเนื่อง
- ใช้เทคนิค **8 symmetries** (หมุน 4 ทิศ × พลิก) เพื่อหาตำแหน่งต่อที่เป็นไปได้
- ใช้กลไก retry ทั้ง chain เพื่อรับประกันว่าวาง block ได้ครบ
- ผลทดสอบ (verified): วางได้ 4/4 และ 5/5 blocks ครบทุก seed
  พร้อมยืนยันว่าเส้นต่อเนื่อง (one-stroke) และไม่มีช่องซ้ำ

### 2.4 Metrics สำหรับประเมินคุณภาพด่าน

- **Crossings** — จำนวนคู่ช่องที่อยู่ติดกันบน grid แต่ไม่ต่อเนื่องกันใน path
  (สะท้อนความซับซ้อนของเส้น)
- **Decision Points** — ช่องที่มีเพื่อนบ้านบน path ≥ 3 ช่อง (จุดที่ต้องตัดสินใจเลือกทิศ)
- **Straightness** — สัดส่วนช่วงที่เดินตรง

## 3. แนวทาง Algorithm ที่สำรวจ (7 แนวทาง)

ตารางสรุปแนวทางทั้งหมดที่ได้พิจารณา พร้อมการประเมิน:

| # | แนวทาง | หลักการ | การันตีว่าด่านแก้ได้ | ปัญหา/ข้อจำกัด | สถานะ |
|---|--------|---------|:---:|----------------|-------|
| 1 | Backbite เต็ม grid | Markov Chain สลับเส้นทาง | ใช่ | ด่านเต็มเป็น blob | ทดสอบแล้ว |
| 2 | Tail-Cutting | ตัดปลาย path เป็น space | ใช่ | ช่องว่างเกาะกลุ่มที่ปลาย | ทดสอบแล้ว |
| 3 | Random Hole Punching | เจาะรูก่อนแล้วหา path | **ไม่** ต้องตรวจสอบ | อาจ unsolvable | ตัดทิ้ง |
| 4 | Chain Topology | ต่อหลาย path ด้วย bridge | ใช่ | จำนวนช่องไม่คงที่ | ทดสอบแล้ว |
| 5 | Longest Path | เจาะ space แล้วหาเส้นยาวสุด | ใช่ | **NP-hard** | ตัดทิ้ง |
| 6 | Random Self-Avoiding Walk | เดินสุ่มจนตัน | ใช่ | ตันเร็ว, retry สูง | เสนอแล้ว |
| 7 | MST-Guided Random Walk | สร้างแผนที่นำทางก่อนเดิน | ใช่ | ต้องปรับให้มี space + trap | กำลังพิจารณา |

### หลักการแกนที่ได้ข้อสรุป

จากการสำรวจ พบว่าแนวทางที่ดีควรยึดหลัก
**"สร้าง path ก่อน → ช่องว่างคือส่วนที่ path ไม่เดินผ่าน"**
เพราะ path ที่สร้างขึ้นคือ "คำตอบ" ในตัวเอง ทำให้**การันตีได้ว่าด่านแก้ได้เสมอ**
โดยไม่ต้องเสียเวลาตรวจสอบย้อนหลัง — ซึ่งตรงข้ามกับแนวทางที่ "เจาะรูก่อน"
ที่อาจสร้างด่านที่ไม่มีคำตอบ

## 4. ข้อค้นพบเชิงทฤษฎีที่สำคัญ

ส่วนนี้คือองค์ความรู้เชิงทฤษฎีที่ได้จากการวิเคราะห์ ซึ่งมีผลโดยตรงต่อการตัดสินใจออกแบบ

### 4.1 ข้อจำกัดของ Hamiltonian Path: degree-1 ได้แค่ 2 จุด

ใน Hamiltonian Path ช่องที่มีเพื่อนบ้านเพียงช่องเดียว (degree-1 — ทางตันแท้)
**จำเป็นต้องเป็นจุดปลายของเส้น (endpoint) เท่านั้น** และเนื่องจากเส้นเดียวมีปลายได้
เพียง 2 จุด (จุดเริ่ม + จุดจบ) ดังนั้น:

> **ด่านที่มีทางตันแท้ (degree-1) เกิน 2 จุด จะไม่มี Hamiltonian Path = แก้ไม่ได้**

ข้อค้นพบนี้สำคัญมาก เพราะเป้าหมาย "อยากให้มี dead end เยอะ" ขัดกับธรรมชาติ
ของ Hamiltonian Path โดยตรง จึงต้องตีความ "dead end" ใหม่ (ดูข้อ 4.6)

### 4.2 ความสมมูลระหว่าง Grid และ Graph

ได้พิจารณาว่าการเปลี่ยนมุมมองจาก "ตาราง (Grid)" เป็น "โหนดและเส้นเชื่อม (Graph)"
จะช่วยลดความซับซ้อนหรือไม่ — ข้อสรุปคือ **ไม่** เพราะหากแต่ละโหนดเชื่อมกับ
เพื่อนบ้าน 4 ทิศแบบ grid มันคือสิ่งเดียวกันทุกประการ (เรียกว่า Grid Graph)

ข้อจำกัดเรื่อง dead end ในข้อ 4.1 ไม่ได้มาจาก "การใช้ grid" แต่มาจาก
**topology ของการลากเส้นเดียว (เส้นมี 2 ปลายเสมอ)** จึงติดตัวไปทุกรูปแบบการนำเสนอ

อย่างไรก็ตาม มุมมองแบบ Graph มีประโยชน์ในแง่ (1) ควบคุม degree ของแต่ละโหนด
ได้ละเอียดกว่า และ (2) นำทฤษฎีกราฟมาใช้พิสูจน์ในเล่มได้ตรงขึ้น

### 4.3 Longest Path Problem เป็น NP-hard

แนวคิด "เจาะ space แล้วให้ algorithm หาเส้นทางที่ยาวที่สุด" ถูกตัดทิ้ง เพราะ
**Longest Path Problem เป็นปัญหา NP-hard** ไม่มี algorithm ที่หาคำตอบที่ดีที่สุด
ได้ในเวลาที่ยอมรับได้บน grid ขนาดใหญ่ การใช้ DFS + backtracking จะระเบิด
(combinatorial explosion) เมื่อ grid โต เช่น 8×8 (64 ช่อง)

### 4.4 เงื่อนไข Parity และ Connectivity

หากเลือกแนวทาง "เจาะรูก่อน" รูปทรงที่เหลือต้องผ่าน 2 เงื่อนไขจึงจะมี Hamiltonian Path:
- **Connectivity** — ทุกช่องที่เหลือต้องเชื่อมถึงกันได้
- **Parity (Checkerboard)** — เมื่อระบายสีแบบกระดานหมากรุก จำนวนช่องสองสี
  ต้องต่างกันไม่เกิน 1 มิฉะนั้นจะไม่มี Hamiltonian Path
  (เช่น รูปแบบ checkerboard ที่ไม่สมดุลจะเป็นไปไม่ได้)

นี่คือเหตุผลที่แนวทาง "เจาะรูก่อน" (#3) ต้องมีขั้นตอนตรวจสอบและสุ่มใหม่ จึงไม่ถูกเลือก

### 4.5 Spanning Tree → Hamiltonian Path (เทคนิค Wall-Following)

ได้ศึกษาเทคนิคคลาสสิกที่ใช้ในการสร้าง Hamiltonian Cycle (พบในงาน AI ของเกม Snake
และการสร้างเขาวงกต/Maze) คือ:
1. ยุบ grid ขนาด N×M ให้เหลือ "coarse grid" ขนาด N/2 × M/2
2. สร้าง Spanning Tree บน coarse grid (ด้วย Kruskal's หรือ Prim's Algorithm)
3. "เดินเลียบรอบ" ต้นไม้ (tree-tracing / wall-following) → ได้เส้นทางที่ผ่าน
   ทุกช่องของ grid เดิมครบครั้งเดียว = Hamiltonian Path/Cycle เสมอ

**ข้อดีสำคัญ:** เทคนิคนี้การันตีว่าได้เส้นเต็มโดยไม่ตันกลางคัน แก้ปัญหา
"การตันก่อนเวลา (Premature Termination)" ของ Random Walk บริสุทธิ์ได้ 100%

### 4.6 Articulation Points และนิยามที่ถูกต้องของ "Trap"

เนื่องจากทางตันแท้ (degree-1) ใส่ได้แค่ 2 จุด (ข้อ 4.1) "dead end เยอะ"
ที่ต้องการจริงจึงต้องตีความเป็น **"ทางตันลวง" (Trap)** — คือสถานการณ์ที่ผู้เล่น
ลากเส้นเข้าไปผิดจังหวะ แล้วเหลือช่องที่เข้าไม่ถึงอีกต่อไป

Trap ที่ถูกต้อง (ที่ไม่ทำลายความเป็น Hamiltonian Path) เกิดจาก **Articulation Point**
— จุดคอขวดของรูปทรงด่านที่ถ้าตัดออกจะแยกด่านเป็นสองส่วน ผู้เล่นต้องเดินผ่าน
จุดนี้ให้ถูกจังหวะ มิฉะนั้นจะเข้าถึงอีกฝั่งไม่ได้

> **ข้อควรระวัง:** การสร้าง trap ด้วยการ "แตกหน่อ/แตกกิ่ง (Sprout)" ออกจากเส้นหลัก
> นั้น**ใช้ไม่ได้** เพราะกิ่งที่ยื่นออกมาจะสร้างโหนด degree-3 (จุดแยก) และ degree-1
> (ปลายกิ่ง) ทำให้โครงสร้างกลายเป็น "ต้นไม้ (Tree)" ไม่ใช่ "เส้นทาง (Path)" อีกต่อไป
> ส่งผลให้ด่าน **ไม่มี Hamiltonian Path = แก้ไม่ได้**

## 5. นิยามเกมที่ตกผลึก (Locked Definition)

ผ่านการอภิปราย ได้ข้อสรุปนิยามที่ชัดเจนและตายตัวว่า:

> **เกม = ลากเส้นเดียวผ่านทุกช่องที่เล่นได้จนครบ โดยไม่ผ่านจุดเดิมซ้ำ → ชนะ**
> **= Hamiltonian Path แท้ (ผ่านทุก *โหนด* ครั้งเดียว ไม่ใช่ทุก *เส้นเชื่อม*)**

นิยามนี้ทำให้ algorithm ทุกตัวที่เลือกใช้ต้องการันตีว่าผลลัพธ์เป็น Hamiltonian Path
ที่มีคำตอบเสมอ และความยากต้องมาจาก "trap" (ข้อ 4.6) ไม่ใช่ทางตันแท้

## 6. ทิศทางการออกแบบปัจจุบัน: MST-Guided (ฉบับปรับปรุง)

แนวทางที่กำลังพัฒนาคือการนำเทคนิค Spanning Tree (ข้อ 4.5) มาปรับให้ได้คุณสมบัติ
ครบทั้ง 3 ข้อ โดยแก้ 2 จุดอ่อนของข้อเสนอ MST ดั้งเดิม:

| ประเด็น | MST ดั้งเดิม | ฉบับปรับปรุง |
|---------|-------------|--------------|
| มีช่องว่าง (space) | ❌ เดินเต็ม grid (เป็น blob) | ✅ ใช้ **Partial Spanning Tree** ครอบแค่บางส่วน |
| การสร้าง trap | ❌ แตกหน่อ (ทำลาย path) | ✅ ตัดกิ่งให้เกิด **คอขวด (articulation)** |
| การันตีแก้ได้ | ⚠️ unsolvable ถ้าแตกหน่อ | ✅ path เป็นเส้นเดียวบริสุทธิ์เสมอ |

### Pipeline ที่ปรับปรุงแล้ว
```
1. ยุบ grid → coarse N/2 × M/2
2. สร้าง Partial Spanning Tree (คุม coverage % เพื่อคุมจำนวน space)
   - ตัดกิ่ง/ใบ เพื่อสร้างคอขวด (= trap)
3. Wall-following รอบต้นไม้ → Hamiltonian Path (ไม่ตัน, retry ≈ 0%)
4. ช่อง coarse ที่อยู่นอกต้นไม้ → กลายเป็น space
5. วัด Trap Score + Articulation จากรูปทรงที่ได้
```

### Metric ใหม่ที่จะเพิ่ม: Trap Score
```
Trap Score = Σ (จำนวนจุดแยก × ความลึกของพื้นที่ปิดเมื่อเดินผิด)
```
แนวคิด: ณ จุดแยกใดๆ ให้โปรแกรมลองเดินไปทางที่เป็น trap แล้วนับว่าเดินได้กี่ช่อง
ก่อนถึงทางตัน — ยิ่งลึก (เช่น 5-6 ช่องค่อยรู้ตัว) ยิ่งเป็น trap ที่ดี เพราะลวงผู้เล่น
ได้แนบเนียนและเสียเวลามากกว่า

## 8. สรุป

ช่วงงานนี้ได้ (1) พัฒนาเครื่องมือทดสอบที่ใช้งานได้จริง และยืนยันว่า Backbite
และ Chain ทำงานถูกต้อง (2) สำรวจแนวทาง algorithm 7 แบบอย่างเป็นระบบ พร้อมประเมิน
ด้วยทฤษฎีกราฟ และ (3) ตกผลึกองค์ความรู้เชิงทฤษฎีที่สำคัญ — โดยเฉพาะข้อจำกัด
